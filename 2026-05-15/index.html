技巧一：给云端 Agent 环境开启「版本历史」，出错直接回滚
这个功能是什么？
Cursor 云端 Agent 现在为每个开发环境维护独立的「版本历史」，你可以随时查看某次配置变更记录，并一键回滚到任意历史版本。企业管理员还可以把回滚权限限制为仅限管理员操作，防止误操作。

怎么用？
进入 cursor.com/dashboard/cloud-agents → 选择环境 → 点击「版本历史」→ 选择目标版本 → 确认回滚。审计日志会记录每一步操作的操作人与时间。

适合什么场景？ 团队 Dockerfile 被人误改、依赖版本升级引发 Agent 运行异常、需要追溯某次配置变更责任的合规场景。

技巧二：用「网络出口白名单」给 Agent 套上安全围栏
这个功能是什么？
云端 Agent 在工作时会访问外部 API、代码仓库、内部服务等。Cursor 现在支持在环境级别限制出站网络访问，你可以为某个环境设置只允许访问特定域名白名单，其他环境保持宽松设置——互不影响。

怎么用？
环境配置 → 「网络策略」→ 选择「限制模式」→ 添加允许访问的域名列表（如 github.com、npm.registry.com、内部 CI 地址）。

适合什么场景？ 金融、医疗等对网络隔离有合规要求的企业；需要防止 Agent 意外调用第三方服务泄露代码的高安全级别团队。

技巧三：「密钥作用域隔离」——给每个 Agent 环境配独立密钥，互不污染
这个功能是什么？
为某个环境配置的 API Key、数据库凭证等密钥，无法从任何其他环境访问——即使同一团队的两个不同 Agent 环境也完全隔离。这比在团队级别共享密钥安全得多。

怎么用？
环境设置 → 「Secrets」→ 添加密钥（名称 + 值）→ 保存后该密钥仅在当前环境的 Agent 运行时可见，其他环境查不到。Dockerfile 构建密钥（--secret）也遵循同样的隔离规则，构建完成后自动销毁。

适合什么场景？ 同一团队维护多套环境（dev/staging/prod）时，防止测试环境密钥越权访问生产数据；外包团队接入时限制其只能访问特定环境。
