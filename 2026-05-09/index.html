今日 Cursor 使用技巧

来源：Cursor 官方博客「用 autoinstall 自举 Composer」· 2026-05-06 发布

技巧一：用「双 Agent 协作法」接手陌生项目——先让 AI 制定验收标准，再开始干活

是什么：不要直接对 Cursor 说「帮我把这个项目跑起来」——先开一个 Composer 对话，让它读 README / Makefile，输出「把这个项目跑通，需要哪些命令和预期结果（3条核心验收标准）」。然后关掉这个对话，再开一个全新的 Composer，把这 3 条标准粘进去，让它去真正搭环境。

  • 接手遗留项目：先问「这个项目跑通的 3 个最小验证命令是什么」，再执行
  • 需求开发：先让 AI 写出「这个功能完成后能看到什么效果 / 哪些测试要过」，存进 CLAUDE.md，再写代码
  • 调试复杂 Bug：先让 AI 描述「正常情况下这里应该输出什么」，再让另一个 Agent 找差异

为什么好用：没有「可验证的成功标准」，AI 容易陷入瞎改循环——改了 A 坏 B，改了 B 又坏 C。把「定义目标」和「执行任务」分给两个独立 Agent，就能大幅降低这种漂移。Cursor 官方用这套方法把 Composer 2 的 Terminal-Bench 得分从 47.9% 提升到 61.7%。

技巧二：让 Cursor 「模拟」缺失的依赖——不完美环境也能跑起来，训练模型的方法你也能用

是什么：接手一个依赖复杂、文档残缺、数据库连不上的烂摊子时，不要卡在「等真实环境就绪」——让 Cursor 用模拟依赖先把代码跑通：

  • 用 MinIO 模拟 S3 存储
  • 用占位假数据代替真实数据库表
  • 写启动脚本模拟 sidecar 容器
  • 图片缺失就用纯色占位图先跑通渲染逻辑

适合场景：新人接手老项目搭本地开发环境 / CI 环境没有完整基础设施 / 快速验证某个模块逻辑不想等运维配环境。核心原则：「能获得反馈信号」比「完全还原生产环境」更重要，先跑通再完善。

技巧三：autoinstall 的「5 次重试上限」思路——给 AI 足够试错空间，又不陷入无效循环

是什么：让 Cursor 自动搭环境或修复某个问题时，建议在 prompt 里明确告诉它：「最多尝试 5 次，每次失败后分析原因换不同策略，5 次还没解决就停下来告诉我卡在哪里」。

为什么：不设上限 AI 会无限重试，消耗大量 Token 却不出结果；设太少（比如 2 次）又不够。Cursor 官方在 autoinstall 中验证过——5 次是效率和容错的最优平衡点。超过 5 次未解决的任务，大多是根本性问题（缺少关键信息 / 权限 / 工具），需要人工介入，AI 继续尝试也无益。
